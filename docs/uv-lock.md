# uv.lock Overview and Potential Hermeto Integration

## Introduction

[uv](https://github.com/astral-sh/uv) is a modern Python package manager developed by Astral.
It provides significantly faster dependency resolution and installation compared to traditional tools such as pip.

Unlike pip, which commonly relies on `requirements.txt`, uv generates a lockfile named **`uv.lock`** that contains a fully resolved dependency graph, including artifact metadata and cryptographic hashes.

This document describes the structure of `uv.lock` files and discusses how they may be integrated into Hermeto in the future.

---

## What is `uv.lock`?

A `uv.lock` file is a lockfile generated by uv after dependency resolution.
It records exact dependency versions and metadata needed to reproduce the environment.

Key characteristics:

* Fully resolved dependency graph
* Exact package versions
* Direct artifact URLs
* SHA256 hashes
* Platform-specific wheels
* Optional dependency markers

Because of this, `uv.lock` already contains most of the information needed for Hermeto to fetch dependencies without performing dependency resolution itself.

---

## High-Level Structure

A simplified `uv.lock` file typically contains:

```
version
revision
requires-python
resolution-markers

[[package]]
name
version
dependencies
sdist
wheels
```

Example:

```
[[package]]
name = "numpy"
version = "2.4.2"

sdist = { url = "...", hash = "sha256:..." }

wheels = [
  { url = "...manylinux_x86_64.whl", hash = "sha256:..." },
  { url = "...macos_arm64.whl", hash = "sha256:..." }
]
```

Each `[[package]]` entry represents a resolved dependency.

---

## Package Metadata

Each package entry may contain:

| Field          | Description                    |
| -------------- | ------------------------------ |
| `name`         | Package name                   |
| `version`      | Exact resolved version         |
| `dependencies` | Direct dependencies            |
| `sdist`        | Source distribution artifact   |
| `wheels`       | Platform-specific wheels       |
| `source`       | Package index (typically PyPI) |

Example dependency entry:

```
dependencies = [
  { name = "numpy" },
  { name = "python-dateutil" }
]
```

---

## Distribution Artifacts

Packages may provide two types of distributions.

### Source Distribution (sdist)

```
sdist = {
  url = "...tar.gz",
  hash = "sha256:..."
}
```

Source distributions must be built locally during installation.

### Wheel

```
wheels = [
  { url = "...manylinux_x86_64.whl", hash = "sha256:..." }
]
```

Wheels are prebuilt binary packages and are platform-specific.

A single package may include many wheels for different platforms and Python versions.

---

## Environment Markers

Some dependencies include environment markers.

Example:

```
{ name = "tzdata", marker = "sys_platform == 'win32'" }
```

This indicates the dependency should only be installed under certain environments.

Markers may depend on:

* operating system
* Python version
* platform architecture

Correct handling of markers is necessary to determine which dependencies apply to a specific target environment.

## Implications for Hermeto

Hermeto currently supports Python dependency fetching using pip requirements files.
Supporting `uv.lock` could provide several advantages.

### Potential Benefits

* Dependency resolution already completed
* Artifact URLs and hashes already available
* Reduced network queries to package indexes
* Faster dependency fetching

### Possible Integration Approach

A future implementation could:

1. Parse the `uv.lock` file
2. Extract packages and distribution artifacts
3. Evaluate environment markers for the target platform
4. Select compatible wheels using Hermeto's existing binary filtering logic
5. Download required artifacts for offline usage

This approach could reuse existing wheel selection mechanisms currently used by the pip backend.

---

## Future Work

Supporting `uv.lock` would require:

* A lockfile parser
* Marker evaluation
* Artifact selection
* Integration with Hermeto’s download pipeline

Further exploration and discussion would help determine the best design for integrating uv support within Hermeto’s architecture.
